# Отчёт по лабораторной работе №2. Hadoop & Apache Spark.

**Цель работы:**
Исследовать, как конфигурация кластера Hadoop и оптимизации Apache Spark влияют на производительность обработки больших данных.
Для этого провести 4 эксперимента:
 * Spark на кластере с 1 DataNode, без оптимизаций
 * Spark на кластере с 1 DataNode, с оптимизациями
 * Spark на кластере с 3 DataNode, без оптимизаций
 * Spark на кластере с 3 DataNode, с оптимизациями

##  Датасет
Датасет был сгенерирован в Jupyter ноутбуке `dataset_create.ipynb` и сохранен в `ecommerce_orders.csv`.

**Тема датасета**

Имитация записей заказов интернет-магазина за один год.
Каждая строка — это одна позиция в заказе.

**Размер датасета**

120,000 строк x 9 столбцов

**Столбцы**
* order_id — уникальный номер заказа, обычный ID.
* order_date — дата и время заказа. Хранится как строка в формате ГГГГ-ММ-ДД ЧЧ:ММ:СС.
* product_category — категория товара (электроника, одежда, товары для дома, книги, спорт). Это категориальный признак.
* quantity — количество товаров в заказе.
* unit_price — цена за единицу товара.
* delivery_region — регион доставки (Москва, СПб, Казань, Новосибирск, удаленные регионы). Второй категориальный признак.
* is_express — флаг экспресс-доставки (да/нет).
* customer_rating — оценка клиента от 1 до 5. В этом признаке мы намеренно оставили около 5% пропусков.
* total_amount — итоговая сумма строки заказа. Вычисляемый признак (quantity умножить на unit_price).

**Типы данных столбцов**
| Столбец | Тип данных |
|---|---|
|order_id  |              str|
|order_date |             str|
|product_category |       str|
|quantity         |     int32|
|unit_price       |   float64|
|delivery_region  |       str|
|is_express       |      bool|
|customer_rating  |   float64|
|total_amount     |   float64|

##  Запуск

**1. Настройка окружения**
Клонировать репозиторий:
```
git clone https://github.com/Arugulaa/Lab-2-Spark-Hadoop.git
cd Lab-2-Spark-Hadoop
```
Создать виртуальное окружение и активировать его:
```
python -m venv .venv
# Для Windows (Git Bash):
source .venv/Scripts/activate
# Для Linux / WSL:
source .venv/bin/activate
```
Установить зависимости:
```
pip install -r requirements.txt
```
**2. Запуск экспериментов**

2.1. Открыть *Git Bash*.

2.2. Перейти в папку проекта (в которую клонировался репозиторий). Пример:
```
cd /С/projects/spark_lab  
```
2.3. Запустить последовательно в *Git Bash* cледующие команды:
```
# 1 DataNode, базовый вариант
bash scripts/run_1dn_base.sh

# 1 DataNode, оптимизированный вариант
bash scripts/run_1dn_opt.sh

# 3 DataNode, базовый вариант
bash scripts/run_3dn_base.sh

# 3 DataNode, оптимизированный вариант
bash scripts/run_3dn_opt.sh

```

##  Анализ результатов
Оптимизация настроек Spark на одном DataNode уменьшила время выполнения с 7.77 s до 5.69 s без заметного влияния на потребление RAM (~27 MB). Масштабирование кластера до трёх DataNode дало ещё больший выигрыш (до 5.10 s), однако при  оптимизации на 3DN время выполнения увеличилось до 6.17 s при том же уровне памяти.
 > На одном узле (1 DataNode) базовая конфигурация создаёт относительно небольшое число партиций, из‑за чего несколько крупных задач обрабатываются с недостаточным параллелизмом, и основное время уходит на этап чтения данных (около 5.8 s из 7.8 s общего времени).
> При оптимизации Spark Application на 1 DataNode (увеличение spark.sql.shuffle.partitions и явный repartition(8)) данные делятся на большее количество более мелких партиций, что позволяет уменьшить время чтения до ~3.6 s, а общее время до ~5.7 s. Потребление RAM остается практически без изменений (~27 MB во всех конфигурациях).
> Далее в 3-м эксперименте мы масштабируем кластер до трёх узлов (3 DataNode), что даёт ещё больший выигрыш:  базовый вариант на 3DN выполняется за ~5.1 s , что примерно в 1.5 раза быстрее, чем 1DN‑baseline. В данном случае за счёт параллельного чтения блоков HDFS на нескольких узлах кластера уменьшается общее время выполнения операций.
> 4-й эксперимент показал, что не всегда попытка одновременно масштабировать кластер до 3 DataNode и дополнительно применять repartition(8) приводит к повышению производительности. Это можно наблюдать по увеличению общего времени до ~6.2s. В данном примере организация большего количества партиций лишь создает дополнительную нагрузку, способствующую увеличению общего времени выполнения операций.

Таким образом, в данной постановке задачи основным фактором ускорения является масштабирование кластера и разумный выбор числа партиций, тогда как "агрессивная оптимизация" параметров shuffle и явный repartition() на небольших датасетах могут не только не помочь, но и заметно замедлить выполнение.
